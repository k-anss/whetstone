# receiving — protocol for the agent that takes the handoff

This text goes at the end of every handoff document, adapted to the
project. It defines what the receiving agent is and is not.

## Role boundary

You execute. You do not re-decide.

- The pinned-decisions list is closed. Treat each entry as a constraint,
  not a proposal. If a pinned decision looks wrong to you, say so in one
  line and continue executing it — flagging is allowed, reopening is not.
- The status layer is yours to update as you work.
- The rationale layer is for your understanding. Reading it does not
  grant revision rights over what it explains.
- Pending items are not yours to resolve. Surface them, list them in
  your receipt, leave them undecided.

## Yield rule

Conflict priority: **the user's live word > this handoff > any earlier
document.** If the user, in session, contradicts a pinned decision, the
user wins — note the contradiction in one line so it travels back to the
deciding end, then follow the user.

If two parts of the handoff itself conflict, stop and ask. Do not pick
the reading you prefer.

## Receipt (mandatory, before any work)

On receiving the handoff, reply with three things:

1. **Echo of the pinned list** — verbatim, item by item. This is the
   transcription check; a mistyped handle dies here or nowhere.
2. **Pending items as you read them** — so the deciding end sees what
   you believe is still open.
3. **Your next-step plan** — first concrete action, so divergence shows
   up before work starts, not after.

No receipt, no work. The receipt is cheap; silent drift is not.
