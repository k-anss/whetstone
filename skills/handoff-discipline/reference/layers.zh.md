# layers — 把材料分进三层

写任何交接前先读本文件。源材料的每一句话,归入三层之一,或归入
危险类。

## 第一层 · 不许重开(已拍死)

已经做出、接手方不得重新评估的决定:命名、账号名、架构选择、范围
裁剪、已关闭的战略问题。

判定问句:**这个若被重开,是浪费还是进步?**
浪费 → 进此层。进步 → 不属于此层。

已拍死项写成编号清单,一行一个决定;凡精确性要紧的(账号名、仓库名、
许可证字符串)原样照抄。精确就是目的:已拍死项存在的意义,是让接手
方能逐字核对,而不是转述。

## 第二层 · 当前状态(可更新)

在跑的、已完成的、排队的。接手方被允许——且被期待——在工作中更新
这一层。

状态行陈述事实,不陈述意愿:写"6 个文件已成 3 个",不写"进展顺利"。
一条接手方无法验证或证伪的状态行,是伪装成状态的理由——挪走它。

## 第三层 · 战略理由(为什么这么定)

策略、取舍、已拍死项背后的推理。接手方读这一层是为了理解,不是为了
修改。它存在的意义,是当接手方撞到交接没覆盖的情形时能正确外推——
不是让它对已覆盖的情形产生二心。

理由层可选。理由层为空的交接仍是合法交接;理由渗进已拍死清单的交接
不是。

## 危险类 · 待重审项

既不是已拍死、也不是普通状态的条目:排期要重审的判断、等某个具名者
拍板的开放问题。

必须显式标注——"待定:等 X 拍板"——并隔离在已拍死清单之外。失败
模式双向都有:待定项误归已拍死,会冻结本该更新的判断;已拍死项误归
待定,等于邀请接手方重开它。

分不清一条归哪边时,不猜。停下来让用户归类。(这就是本 skill 声明的
失败行为;它最常在这里被触发。)
