# Handoff · whetstone skill build → fresh executing session

## Project in one line
A skills monorepo (`whetstone`, account `k-anss`, MIT, bilingual EN/ZH)
where each skill encodes one domain judgment as an agent-executable
discipline file.

## Pinned decisions (do not reopen)
1. Skill name: `handoff-discipline`. Alternatives were considered and
   rejected at the deciding end.
2. Repo: `whetstone`, monorepo, skills under `skills/`. Account: `k-anss`.
   License: MIT. Author byline: K.
3. Structure: SKILL.md + 3 reference files (layers / template / receiving)
   + 1 example, each with a `.zh` twin.
4. SKILL.md description text: fixed verbatim at the deciding end,
   including negative triggers.
5. One session builds one skill, single pass. Rework requests are logged
   and carried back to the deciding end — not fixed in-session.

## Strategic rationale (read to understand, not to revise)
The skill's differentiation is judgment density, not feature count. All
rule-thickening material (mechanism / observation / boundary) must come
from the author's real practice; searching the web for "best practices"
filler is prohibited because it erases provenance.

## Red lines
- No real employer names, no city/county-level place names, no real
  personal names, no traceable work details. Example data is fictional
  or sanitized.
- No emoji, no engagement-bait copy.

## Receiving agent: role and receipt
You execute the build per the pinned structure. You do not rename, do not
restructure, do not re-litigate the description text. Receipt before work:
echo the pinned list verbatim, list pending items as you read them, state
your first concrete action.

## Conflict priority
User's live word > this document > any earlier document.
