# handoff-discipline(跨上下文交接纪律)

摘要丢掉的,恰是下一个 agent 不许重新决定的东西。本 skill 把一个会话
的工作状态,变成下一个 agent 可以直接执行、而不重开已关闭问题的交接
文档。

## 立场(承重偏见)

**1. 交接是决定边界的转移,不是信息的转移。**
机制:摘要保留信息、丢弃决定状态。接手方更需要的是状态——哪些问题
已经关闭——因为分不清"已关 / 未关"的 agent,会满怀信心地重审已关闭
的问题。
实地观察:一次跨实例交接中,一个账号名被手误写成了字序颠倒的变体。
交接文档没带"已拍死决定清单",也不要求逐项回执,接手实例就照错的
往下建,流水线里没有任何环节拦它——错误一路静默传播进了产出。
验证边界:在"同一人、同一长期项目、两个 AI project 之间"的交接上
验证过。跨人、跨组织的交接:未测。

**2. 三层必须显式分离:不许重开 / 当前状态 / 战略理由。**
机制:三层混排时,接手方分不清哪句是固定约束、哪句是可更新的现状。
战术状态会被读成战略邀约;已拍死的决定会被读成"可以改进的背景"。
实地观察:同一套交接实践里,好用的文档都是把已拍死决定编号列在顶部、
与进度记录分开的那种;混排的那种,会产出"热心"重新设计已定结构的
接手方。
验证边界:层分离在项目状态类交接(命名、架构、范围决定)上用过。
纯创作草稿的交接:未测——那里"不许重开层"可能近乎空。

**3. 接手方要有成文的角色边界,含退让规则。**
机制:没有显式的"执行、不重新决定"条款和冲突优先级(用户当场说法 >
交接文档 > 更早文档),接手方会退回它的通用乐于助人先验——其中就
包括重新决定。
实地观察:源实践中带回执要求(回显已拍死清单 + 待定项 + 下一步)的
交接,在入口就拦下了誊写漂移,而不是等到交付才发现。
验证边界:退让规则只在"两端是同一个用户、可当场裁决"的情形下测过。

## 核心动作

1. **先分类再动笔。** 把源材料分进三层——不许重开 / 当前状态 / 战略
   理由——并单独隔离危险类:待重审项,它既不属于已拍死也不属于普通
   状态。判定规则:读 `reference/layers.zh.md`。
2. **套模板。** 用固定字段顺序的交接骨架。字段说明:读
   `reference/template.zh.md`。
3. **附接手方协议。** 角色边界、退让规则、回执要求。正文:读
   `reference/receiving.zh.md`。
4. **产出前过闸。** 把"已拍死决定清单"回显给用户确认,确认后才产出
   文档。这是唯一能防止"待定项被冻成已定项"的环节。

## 六字段

- **版本**:0.1(2026-06)。changelog:修正只增不删;被取代的条款打
  反向标记,旧交接文档归档不覆盖。
- **废止条件**:当 AI 系统具备跨会话、跨实例的持久共享记忆,交接不再
  需要人工文档时,本 skill 退役。
- **失败行为**:源材料分不清"已拍死 / 待定"时,停下来让用户逐项归类。
  不替用户猜哪些已拍死——猜错要么冻结本该更新的判断,要么重开不该
  重开的决定,两种代价都是静默的。
- **组合**:可与任何项目管理 / 记忆类 skill 组合;本 skill 只管"跨
  无上下文实例"的交接结构。遇用户显式指令,退让。
- **时间**:判断定于 2026-06。"AI 无持久跨会话记忆"这一前提在变,
  预期 12–24 个月内重审。
- **边界**:在"同一人、同一长期项目、自己两个 AI project 之间"的交接
  上验证过。跨人、跨组织、对抗性交接:未测,不做主张。
