读,不装饰

这个 skill 做什么——由它自己排版的一篇说明。

read-not-decorate · 2026 · 自指演示

这一页是装配出来的,不是设计出来的。文字以一篇写完的 markdown 进来; skill 选了模式,在固定的视觉系统下排版,只在文字本来就有结构的地方调了组件。 这里没有一句话是为了"更好读"被改写的。这份克制就是全部产品。

01活是排版,不是抢救

多数文档工具在做一件事:把撑不住的文字打扮得撑得住——渐变标题、随手拎 一句做成大字引用、每个小标题旁配一个图标。装饰是个信号:有人断定这段文字 自己留不住注意力,于是用颜色把它架起来。

这个 skill 拒绝这个动作。它假设文字自己配得上被读,它的任务只是让开路: 一页暖白、一种墨色、一个只落在判断和结构上的强调色。

如果一篇东西需要装饰才能被读,要修的是文字,不是 CSS。
看到结构才调组件,绝不为凑数而调。 read-not-decorate · 立场第 2 条

02两种读者,一次分叉

排任何东西之前,skill 先选一种模式——因为两种读者对同一篇文字,要的是 相反的东西。

留档
读者是几个月后的作者
保留推理、每个判断的依据、当时真实存在的不确定。为反复 翻查而建。
分发
读者是陌生人
保留能独立站住的干净结论。删掉内部修订痕迹和把握度 标记。

这个分叉不是同一文件上的滑块。它决定哪些组件存活、文件怎么交付。 选晚了,等于选两次。

03每个组件都是一笔债

每加一个组件,就是在宣称这段文字有那种结构。结构不在而硬加,读者会摸到 那条接缝。所以默认是普通段落,倾向永远是更少的组件,不是更多。

边界与不适用
这个 skill 面向写完的长文。简短笔记、原始碎片、聊天记录、 幻灯片、仪表盘都在边界之外——它们需要重做内容,不是排版内容。遇到这类 输入,正确的回应是说明并停手,不为产出 HTML 而产出 HTML。

装饰被去掉、不需要的组件被拒绝之后,剩下的,是一篇凭句子本身的力气 请求被读的文档。这是这个 skill 唯一想守住的东西。
