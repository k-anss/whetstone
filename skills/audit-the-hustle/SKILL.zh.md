# audit-the-hustle · 审查吆喝

你无法核实一个人是否真在赚钱。你只能读信号——所以输出带把握度的信号,永远不给定论。

## 立场

三条规则。每条带机制、一条实地观察、已验证边界。用户的显式指令优先于本文件任何规则——标注偏离,继续执行。

### 1. 给信号加把握度拆分,不给定论

规则:每次审查的结论必须是"X% 像真在经营,Y% 像表演/卖课漏斗"的形式。永远不输出"靠谱/骗局"这种二元定论。

机制:决定性的事实——真实付费流水、销量是否自演——在水面之下,任何外部手段都查不到。对不可验证的事实下二元定论,等于制造虚假确定。校准过的百分比携带二元定论销毁掉的那个信息:证据实际够到了哪里。

实地观察:作者最初做这类审查时,想要的就是"这人到底真不真"的是非答案,然后看着这个诉求一次次产出自信的错误答案。在对多个公开记账经营者的反复审查里,唯一经得起后续证据检验的输出,是百分比拆分。

边界:在"公开声称的网络收入"(独立产品、课程、newsletter)上验证过。证据强时把握度可以很高——但永不给 100% 或 0%。

### 2. 框架够不着的地方,直接明说

规则:只有圈内实时信息才能回答的问题——真实付费、自演、账号背后到底是谁——给一句显式的"在水面之下,本框架够不着",不给硬凑的估计。

机制:模型不在场。有些真相只以圈内实时反馈的形式流通。把信号清单硬套上去,会产出一个看起来严谨、底下空无一物的数字——比没有数字更糟,因为它会顶掉读者自己的圈内直觉。

实地观察:作者见过"创始人是不是在自演销量"这类问题,硬走清单后产出了一个建在空气上的自信数字。每次站得住的动作都是停下来交还:"这条在水面之下——靠你的圈内判断,不靠我。"

边界:在网络经营者真实性审查上验证过;作为通用认识论规则未测。

### 3. 反偏误自查每次必跑,在结论之前

规则:写结论之前,先回答 reference/anti-bias.md 的四个问题,并把结论里每一行标注为"清单触发"或"自行推理"。

机制:模型的默认倾向——名气大=可信、叙事好=真实、机会值得美化——会系统性抬高真实性评分。两个反复出现的失效模式:锚定在头部明星名字上;悄悄用推理顶替证据,却以同等权重呈现两者。

实地观察:在不自查的审查里,作者反复抓到"知名名字被当成'谁是真的'的默认答案"。这个偏误从不以错误陈述的形式出现;它出现的形式是——哪些证据从来没被要过。

边界:四个问题从真实审查中反复出现的失效里提炼;覆盖那些反复出现的偏误,不覆盖所有偏误。

## 工作流

1. 扫五类信号 → 读 `reference/signals.md`。每类标:有 / 无 / 不可验证。
2. 起草结论 → 读 `reference/confidence.md`,按必填格式和水面上/下边界写。
3. 出口闸门 → 跑 `reference/anti-bias.md`。结论每行标"清单触发"还是"自行推理";两者可信度不同,读者必须看得到区别。

完整的全虚构示范:`examples/sample-audit.md`。

## 维护字段

- **版本**:0.1(2026-06)。changelog 纪律:修正就地标注,不删除。
- **废止条件**:当出现可信的、公开的第三方网络收入核验服务、人工尽调不再需要时,本 skill 退役。
- **失败行为**:若问题只能靠水面下信息回答,停下并声明"框架够不着"——不为了显得有用而硬凑把握度。硬凑的数字比没有数字更危险。
- **组合声明**:可与任何调研/搜索类 skill 组合。本 skill 只管真实性审查,不做财报核验。用户显式指令优先。
- **时间字段**:判断定于 2026-06。信号渠道(独立开发者论坛、付款页截图、MRR 面板)随平台演进,预期 12–18 个月重审。
- **已验证边界**:审查"公开声称在网上赚钱的人"的公开真实性。不适用上市公司财报。不适用对私人的背景调查(隐私)。
