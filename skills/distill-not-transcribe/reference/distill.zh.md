# reference/distill.zh.md — 蒸馏线

闸门通过、用户确认结构后才加载。输入:单一作者写给自己用的结构化
方法文档(协议、SOP、作业指导)。

## 七类变换(文档 → skill)

工作文档骑在作者与其常用助手的共享上下文上;skill 的读者是零上下文
的陌生 agent。七类全做,多数失败来自漏做某一类。

| # | 变换 | 操作 |
|---|---|---|
| 1 | 改指"你" | 文档里的"你"=懂作者的助手。重写为一无所知的 agent:依赖私人上下文的引用,删除或自解释。 |
| 2 | 默契显性化 | 作者"显然会做"的任何事,不写成规则就等于对 agent 不存在。 |
| 3 | 立场抽取 | 观点进 agent 层成为硬约束;地方色彩与渠道 grounding 进人类层(README/语言孪生层)。 |
| 4 | 禁区不软化 | 作者的"不要"清单原样继承。差异化几乎全在禁区;软化一分,就向通用 skill 退一分。 |
| 5 | 作者痕迹分拆 | 身份痕迹(谁/哪里/什么单位)→ 通用化或删除;判断痕迹(为什么有这条、什么失败过)→ 原样保留。最常见错误是两类一起删或一起留。 |
| 6 | 加工作流闸门 | 插入"先报结构再写文件"检查点。文档信任读者,skill 不得信任执行者。 |
| 7 | 每条硬规则增厚 | 见下。 |

## 第七类:增厚(规则 → 承重规则)

每条硬规则四件套出厂:

> **规则 + 机制(why)+ 一条脱敏 field 观察 + 验证边界**

- **机制**:带机制的规则能外推到作者没见过的输入;裸规则在那里
  碎掉。"不要 emoji"单摆着薄;补"装饰符号在留档场景注入虚假情绪
  温度,且经 AI 滥用已成低质信号"即可外推。
- **观察**:"这条规则存在,因为我看见 ___ 在 ___ 时死掉"。平淡的
  物理细节,不要戏剧——戏剧化读起来像编的,平实具体读起来像在场。
- **边界**:写明实际验证过的范围,其余默认未测。诚实边界比伪造
  覆盖厚,且 agent 只能对申报过的边界做精确路由。
- **禁走的捷径**:禁止搜"业界最佳实践"填充增厚——那是 slop 的
  生产方式,每段外来填充都在稀释让这个 skill 配存在的判断密度。
  三种材料全部来自作者。作者对某条规则没有观察 → 该条薄着出厂
  并标明薄,不编。

证据来源规则:`reference/evidence.zh.md`。

## 界面纪律

**实质厚、界面薄。** SKILL.md 保持地图(目标 ≤150 行):观察、
机制、边界沉入 reference;地图只放每次都用的,外加一张
"做 X 时加载 Y"表。

## 散料(边界:尚未验证)

结构化协议与散料(对话记录、零散笔记)是两种输入分布。协议的
蒸馏是变换;散料先需一道提取工序(从几万字里捞出"遇 X 选 A
不选 B"的判断,才有东西可变换)。本 skill 只在协议上跑过。
输入是散料时:明说,把提取工序当实验跑,产出照实标注,
并预期本节在首次真实使用后被修订。
