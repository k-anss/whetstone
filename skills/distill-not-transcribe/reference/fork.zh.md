# reference/fork.zh.md — 改编线

存在可比原作时加载。改编高赞 skill 是正当的,常比从零造更好:
高赞原作=已验证需求,且 fork 继承原作名字的搜索引力。
把改编和誊写分开的只有一样东西:**判断增量**。

## 四问

四问全答才动手。任一问答不出 → 不做——那是誊写(或汉化),
没有人有理由弃原版选你。

1. **原作具体在哪失灵?** 点名具体的人的具体动作——不是
   "对某些人可以更好"。
2. **你当过那个人吗?** 失灵必须是亲历的不是想象的:
   "我在真实项目里跑过,它碎在 ___。"
3. **修补是什么、来自哪次真实使用?**
4. **这个修补对原作用户群里的哪一部分人构成换用理由?**

## 测试痕迹页(每个改编必带)

"我测过"是零信用的形容词——每个誊写者都这么写。
**痕迹本身才是指纹。** 每个 fork 的 reference/ 里带一页:

```
## 测试痕迹 — <原作名> v<版本>,测于 <日期>
- 输入:<实际喂了什么,脱敏到结构层>
- 碎在哪:<观察到的行为,平实细节>
- 修补:<确切改动>
- before/after:<两份输出并排或链接>
- 未覆盖:<本次痕迹没测到的>
```

发布前按作者红线标准脱敏。背后没有真实测试的模板就是又一个
slop 模板:还没真测过 → 这个 fork 还没准备好。

## 谱系申报

- `derived_from:` 写明原作、链接、版本。MIT/CC0 下 fork 合法;
  掉信用的是隐瞒——出处可反向核验,被发现的暗 fork 降权整仓。
- 谱系行旁边一句话写清增量:"与 <原作> 的差异在:___"。
  这句难写 → 回四问。
- 继承的内容保持原作精神;不许为制造距离而装饰性重命名概念
  (装修偏见)。
