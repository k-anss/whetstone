# report-deck-zh — 中文汇报有自己的语法,把它说出来

中文工作汇报有一套结构语法——对仗标题、总分总、带口径的数据页——
英文 slide 工具不会说这门语法。本 skill 把它编码下来。

## 谱系(改编线 skill——判它冗余之前先读这节)

derived from / inspired by **frontend-slides**(Zara)。改编四问:

1. **原作在哪失灵?** frontend-slides 做的是英文审美的 HTML slides,
   它不懂中文正式汇报的约定:标题要对仗、全篇要总分总收口、数据页
   要带口径说明、层级按中文公文递进逻辑走,不是自由 bullet。
2. **作者当过失灵用户吗?** 当过。作者做过大量中文正式工作汇报;
   用英文审美工具排出来的 deck,在该场景下结构就是错的——还没读
   一个字,就被认出"不像正式材料"。
3. **修补是什么?** 把英文审美模板换成中文汇报结构语法库:对仗标题
   生成规则、总分总骨架、数据口径页式样、层级递进规则。视觉令牌
   复用 read-not-decorate 的克制系统。
4. **对谁构成换用理由?** 要做中文正式工作汇报(述职/总结/立项/复盘)
   的人——他们要的是符合中文汇报习惯的结构,不是英文 pitch deck 美感。

## 立场(三条承重判断)

1. **中文正式汇报有自己的结构语法,英文 slide 工具不会说。**
   机制:对仗标题、总分总、口径页是这个文体的约定;套英文 deck 逻辑,
   结构本身就错,再精致也没用。现场观察:用英文审美工具排中文汇报,
   标题不对仗、缺数据口径页,有经验的读者一眼判"不像正式材料"。
   边界:在正式工作汇报上验证;学术/政府出版物格式未测。
2. **只做结构,绝不碰内容。** 机制:骨架可通用,内容因人因事,生成
   内容既越权又有泄露真实细节的风险。现场观察:作者自己的生产规则
   反复划的就是这条线。输入不足以判断结构时,停下来问汇报类型——
   不许编内容去填骨架。
3. **克制视觉,复用 read-not-decorate 令牌。** 机制:汇报要专业不要
   花哨;大众 AI-PPT 的同质化卡片美化正是要避开的失败模式。现场观察:
   同质化美化让所有汇报长一个样;克制的结构化版式反而显专业。
   边界:此判断只针对正式汇报;轻量/趣味 deck 合理地需要另一种视觉,
   不在本 skill 范围。

## 核心动作(做 X → 加载 Y)

| 任务 | 加载 |
|---|---|
| 定汇报骨架(类型、章节、收口) | `reference/structure.zh.md` |
| 生成或修复对仗标题 | `reference/titles.zh.md` |
| 排带口径说明的数据页 | `reference/data-pages.zh.md` |
| 套视觉令牌 | `reference/visual.zh.md` |
| 看完整成品示例 | `examples/demo.zh.html` |

**工作流闸门(必走):** 排版前,先向用户报告 (a) 识别出的汇报类型
(述职/总结/立项/复盘/其他)和 (b) 拟定的章节骨架,等确认。骨架未
确认就排版,是最常见的失败。

**硬规则:**

- 同级标题组必须对仗:字数对齐、结构对称、动宾一致
  (`reference/titles.zh.md`)。
- 凡出现数字的页必须带口径说明:数据来源、统计范围、时间口径
  (`reference/data-pages.zh.md`)。没有口径,不成数据页。
- 全篇收口:开篇述总、中间分项、结尾回到总并给出增量
  (`reference/structure.zh.md`)。
- 禁 emoji、禁渐变、禁装饰性图标、禁 stock 配图
  (`reference/visual.zh.md`)。
- 示例与占位内容只用虚构的通用商业场景。

## 六字段

- **版本**:0.1(2026-06)。changelog:首发。修正在原处反向标记,
  不删除。
- **废止条件**:当裸模型在无提示情况下,能稳定为中文汇报产出对仗
  标题、总分总结构和口径说明时,本 skill 退役。
- **失败行为**:输入不足以判断汇报类型或骨架时,停下来问。不静默
  硬套骨架,不用编造内容来补。
- **组合**:composes-with read-not-decorate(共用视觉令牌)。与
  frontend-slides 分工:英文 deck 用那个,中文正式汇报用本 skill。
  用户显式指令优先于本 skill。
- **时间**:判断定于 2026-06。中文汇报习惯稳定,AI-PPT 工具生态不
  稳定,约 18 个月重审。
- **边界**:在中文正式工作汇报版式上验证。不适用英文 pitch deck、
  不适用轻量/社交 slides、不生成汇报内容。此外未测——按未验证对待。
